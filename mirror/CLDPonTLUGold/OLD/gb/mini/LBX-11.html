11. Alternatives

如果你有某些原因不喜欢 lbxproxy 。例如你不喜欢他的执行效率， 它不帮你工作， 你不想麻烦的为远端建立 lbxproxy 或著你只是想是是其他的选择， 这边有一些其他的 X protocol 压缩方式。 (有没有人有其他的方案？)

11.1 dxpc - 一个极不一样的 X Protocol 压缩程式

  • 原本的作者∶ Brian Pane <brianp@cnet.com>
  • 现下的维护者∶ Zachary Vonler <lightborn@mail.utexas.edu>

dxpc 和 LBX 以相同的方式运作。然而为了避免修改 X Server的原始码 以及实作一个 X Server 的扩充dxpc 使用 两个 代理伺服器∶ 一个像是 lbxproxy 跑在远端的系统，另一个则在本端执行。

在远端的代理伺服器负责沟通 X Client 以及本端的代理伺服器， 而本端的代理伺服器负责沟通 X Server 以及远端的代理伺服器。

所以，对於 X clients 以及 X server 两造，他们都以平常的处理方式处理 X Protocol

优点

  • 既然它是完全的独立程式而不需要动到 X 的内部， 它一定非常的 容易编译以及安装。
  • 它是独立的被维护著，所以你不需要 OSF 释出新的 X ，即可过的效能的增进以及错误的修正。
  • 它提供了比 lbxproxy 更多的压缩资讯以及统计资讯。

缺点

  • 它不是 X 的标准套件，你必须自己取得以及编译安装。
  • 它在安装上变的比较复杂，因为它要求在本端也要有一个像是远端的代理伺服器。

我在哪里可以得到 dxpc？

dxpc 的原始档可以自 ftp.x.org取得。

这里有一个网页 http://ccwf.cc.utexas.edu/~zvonler/dxpc/有 dxpc 很多不错的资讯。包含了 dxpc 的 mailing list，原始档，以及 为多种平台预先编译好的执行档。

11.2 Ssh (Secure Shell)

Ken Chase <lbxhowto@sizone.org> notes that ssh 能被用於压缩。虽然它只要的目的是为了保密，但是它也压缩它传送的资料。

所以如果你自一个 ssh 连结执行 X ，你在使用间自然会压缩资料。

11.3 谁比较好？

我不知道。 LBX 以及 dxpc 在纯压缩上比 ssh 好。当然， ssh 提供了在安全上的附加功能。 不过，也没有任何理由禁止你使用 ssh 加上另外两种之一的压缩方式，以获得良好的压缩以及安全性。

在这些选择上跑一些评量以获取统计资讯，应该不是很难的事。 但是我没有做过这件事，也没有听说过任何人做过。
