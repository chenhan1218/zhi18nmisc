11. Alternatives

如果你有某些原因不喜歡 lbxproxy 。例如你不喜歡他的執行效率， 它不幫你工作， 你不想麻煩的為遠端建立 lbxproxy 或著你只是想是是其他的選擇， 這邊有一些其他的 X protocol 壓縮方式。 (有沒有人有其他的方案？)

11.1 dxpc - 一個極不一樣的 X Protocol 壓縮程式

  • 原本的作者︰ Brian Pane <brianp@cnet.com>
  • 現下的維護者︰ Zachary Vonler <lightborn@mail.utexas.edu>

dxpc 和 LBX 以相同的方式運作。然而為了避免修改 X Server的原始碼 以及實作一個 X Server 的擴充dxpc 使用 兩個 代理伺服器︰ 一個像是 lbxproxy 跑在遠端的系統，另一個則在本端執行。

在遠端的代理伺服器負責溝通 X Client 以及本端的代理伺服器， 而本端的代理伺服器負責溝通 X Server 以及遠端的代理伺服器。

所以，對於 X clients 以及 X server 兩造，他們都以平常的處理方式處理 X Protocol

優點

  • 既然它是完全的獨立程式而不需要動到 X 的內部， 它一定非常的 容易編譯以及安裝。
  • 它是獨立的被維護著，所以你不需要 OSF 釋出新的 X ，即可過的效能的增進以及錯誤的修正。
  • 它提供了比 lbxproxy 更多的壓縮資訊以及統計資訊。

缺點

  • 它不是 X 的標準套件，你必須自己取得以及編譯安裝。
  • 它在安裝上變的比較複雜，因為它要求在本端也要有一個像是遠端的代理伺服器。

我在哪裡可以得到 dxpc？

dxpc 的原始檔可以自 ftp.x.org取得。

這裡有一個網頁 http://ccwf.cc.utexas.edu/~zvonler/dxpc/有 dxpc 很多不錯的資訊。包含了 dxpc 的 mailing list，原始檔，以及 為多種平台預先編譯好的執行檔。

11.2 Ssh (Secure Shell)

Ken Chase <lbxhowto@sizone.org> notes that ssh 能被用於壓縮。雖然它只要的目的是為了保密，但是它也壓縮它傳送的資料。

所以如果你自一個 ssh 連結執行 X ，你在使用間自然會壓縮資料。

11.3 誰比較好？

我不知道。 LBX 以及 dxpc 在純壓縮上比 ssh 好。當然， ssh 提供了在安全上的附加功能。 不過，也沒有任何理由禁止你使用 ssh 加上另外兩種之一的壓縮方式，以獲得良好的壓縮以及安全性。

在這些選擇上跑一些評量以獲取統計資訊，應該不是很難的事。 但是我沒有做過這件事，也沒有聽說過任何人做過。
